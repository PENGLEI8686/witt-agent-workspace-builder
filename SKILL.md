# Witt-AgentOS搭建器（Witt Agent Workspace Builder）

扫描用户的工作目录，理解业务上下文，生成一套结构化的 AI 协作项目文件夹。
用户拿到的不是空骨架，而是填好了内容的半成品——已有文件已归位，缺失的文件有带占位提示的模板。

前置条件：Agent 必须具备文件系统读写能力和联网能力。

## 语言策略

- 默认使用用户输入的语言回复。用户用中文提问时用中文；用户用英文提问时用英文。
- 当用户要求双语交付时，同时提供中文和英文版本。
- 文件和目录名默认遵循用户工作区已有语言；`AGENTS.md`、`CLAUDE.md`、`context/`、`skills/`、`memory/`、`output/` 等结构名保持原样。
- 面向英文用户时，将“AI项目工作区文件夹”解释为 local AI agent workspace 或 AgentOS workspace。

---

## 执行流程总览

```
阶段零：指定目录 + 自动扫描
    ↓
阶段一：确认式信息采集（1-2轮对话）
    ↓
阶段二：生成项目文件夹（创建 + 归位 + 模板 + 建议）
    ↓
阶段三：验证 + 后续引导
```

每个阶段完成后向用户确认再进入下一阶段，不要一口气跑完。

---

## 阶段零：指定目录 + 自动扫描

### 步骤 0.1 获取路径

向用户询问：

> 请指定你日常工作最常用的文件夹路径——这个文件夹里通常包含你的业务资料、
> 工作文档、已有的提示词或 AI 配置文件。
>
> 如果你是从零开始、还没有现成的工作文件夹，告诉我也没问题，我会换一种方式引导。

如果用户表示没有现成目录 → 跳转「冷启动路径」（见本文档末尾）。

### 步骤 0.2 安全扫描

对指定目录执行结构扫描，遵循以下规则：

**扫描策略（控制深度和范围）：**
- 先扫目录树结构（仅目录名和文件名，不读内容），最大深度 3 层
- 如果目录包含超过 500 个文件，只扫描根目录 + 一级子目录的结构，不深入
- 对识别出的关键文件做内容抽样（每个文件只读前 50 行），最多抽样 20 个文件

**必须排除的路径和文件（安全规则）：**
- 环境与密钥：`.env`、`.env.*`、`credentials`、`secrets`、`*.pem`、`*.key`、`id_rsa`、`.ssh/`
- 认证与令牌：文件名包含 `password`、`token`、`api_key`、`apikey` 的文件
- 系统与依赖：`node_modules/`、`.git/`、`__pycache__/`、`.venv/`、`venv/`
- 缓存与临时：`.cache/`、`.tmp/`、`*.log`、`.DS_Store`

**关键文件识别（优先抽样阅读）：**
- Agent 配置文件：`CLAUDE.md`、`AGENTS.md`、`AGENT.md`、`.cursorrules`、`system_prompt.*`
- README 和说明文件：`README.md`、`README`、`说明.md`
- Skill 文件：任何名为 `SKILL.md` 的文件及其所在文件夹
- 最近修改的文档：按修改时间排序，取最近 10 个 `.md`/`.txt`/`.docx`/`.pdf` 文件

### 步骤 0.3 形成初步画像

基于扫描结果，整理出：

1. **业务推断**：根据文件命名、目录结构、文档内容片段，判断用户大致在做什么行业/什么工作
2. **资产清单**：
   - 可能适合放入 context/ 的知识文档（按主题初步分类）
   - 可能属于产出物的文件（方案、报告、周报、脚本等——判断标准：这个文件是"参考资料"还是"工作成果"？前者归 context/，后者归 output/）
   - 已发现的 Skill 文件
   - 已发现的 Agent 配置文件
   - 疑似提示词文件（内容符合 System Prompt 或工作流指令的格式，可转化为 Skill）
3. **文件格式统计**：Markdown 文件数量 vs 非 Markdown 文件（.docx/.pdf/.pptx/.xlsx 等）数量
4. **缺失识别**：基于业务推断，这个工作场景通常应该有但扫描中没发现的内容（如：看起来是品牌咨询但没有品牌简介文件）

## 阶段一：确认式信息采集

基于阶段零的扫描画像，用确认式对话而非开放式提问来采集信息。

### 第一轮：业务画像确认

向用户展示初步画像，格式如下：

> **根据你的文件结构，我的判断是：**
>
> - 你的工作领域是 [行业/领域]
> - 你的角色是 [角色描述]
> - 你服务的对象是 [客户/用户描述]
> - 你的核心工作包括 [工作类型列表]
>
> **我找到了这些资产：**
> - [N] 个知识文档，涵盖 [主题列表]
> - [N] 个已有 Skill 文件
> - [N] 个已有 Agent 配置文件
>
> 是否准确？有什么需要补充或纠正的？

用户确认或纠正后，评估是否需要第二轮。

### 第二轮（仅在需要时）：补充采集

以下信息无法从文件扫描中获取，如果第一轮中用户没有主动提及，需要补充询问。
不要一次性全部问完——只问对生成 AGENTS.md 必要的、且确实缺失的内容：

- **合规红线**：你的行业或业务有没有绝对不能触碰的底线？（法规限制、客户隐私规则、行业禁区等）
- **团队协作**：这个项目文件夹是你一个人用，还是团队共用？如果团队用，哪些角色会参与？
- **输出偏好**：你对 AI 输出有什么风格偏好？（语言、格式、详细程度、沟通方式）

### 非 Markdown 文件处理

如果阶段零检测到非 Markdown 文件（.docx/.pdf/.pptx/.xlsx 等），在本阶段向用户展示清单并提供选择：

> **我发现了 [N] 个非 Markdown 格式的文件：**
>
> | 文件 | 格式 | 建议 |
> |------|------|------|
> | [文件名] | .docx | 可转换为 Markdown |
> | [文件名] | .pdf | 建议原样保留 |
> | [文件名] | .xlsx | 建议原样保留 |
>
> 对于每类文件，你希望：
> 1. **转换为 Markdown**（AI 解析效果最好，但可能丢失复杂排版）
> 2. **原样复制**（保留原格式）
> 3. **跳过**（不纳入项目文件夹）

## 阶段二：生成项目文件夹

### 步骤 2.1 确认创建路径

询问用户：项目文件夹创建在哪个路径？建议使用 `[用户工作目录同级]/[项目名称]` 作为默认推荐。

### 步骤 2.2 创建目录结构

```
[项目名称]/
├── AGENTS.md
├── CLAUDE.md
├── context/
│   └── [按主题分的子目录]/
├── skills/
│   └── [已有 Skill 的子目录]/
├── memory/
│   ├── MEMORY.md
│   ├── 决策记录/
│   └── 经验积累/
└── output/
    └── [按产出类型分的子目录]/
```

context/ 的子目录划分基于阶段零的主题分类结果。
output/ 的子目录划分基于用户的核心工作类型。

### 步骤 2.3 生成核心文件

生成以下文件，结构模板见 `references/templates.md`：

1. **AGENTS.md** — 根据采集的信息填充三个区域（身份定义 / 协作规范 / 红线），
   附上知识文件索引表和 Skill 索引表
2. **CLAUDE.md** — 垫片文件，内容固定（见模板）
3. **MEMORY.md** — 初始模板，结构已搭好，内容为占位提示

### 步骤 2.4 归位已有文件

将阶段零识别的文件按以下规则复制（不是移动）到项目文件夹：

| 文件类型 | 目标位置 | 处理方式 |
|---------|---------|---------|
| 知识文档 | context/[对应主题]/ | 复制，按用户选择决定是否转换格式 |
| 产出物 | output/[对应类型]/ | 复制。方案、报告、周报、脚本等工作成果归此处 |
| Skill 文件 | skills/[Skill名称]/ | 复制整个 Skill 文件夹（包括 references/ 子目录） |
| 提示词文件 | skills/[转化后名称]/ | 转化为标准 SKILL.md 格式后放入（见步骤 2.6） |
| Agent 配置 | 项目根目录 | 不复制——已有配置仅作参考，AGENTS.md 是新生成的 |

知识文档 vs 产出物的判断标准：如果这个文件的内容在未来的工作中会被反复参考（品牌介绍、服务手册、组织架构），它是知识文档。如果这个文件是某次具体工作的产出（某期周报、某个活动方案、某篇文章），它是产出物。

复制前向用户展示完整的文件归位清单，获得确认后再执行。

### 步骤 2.5 生成缺失文件的模板

对于阶段零识别的"应有但缺失"的知识文件，在 context/ 对应位置生成带占位提示的骨架文件：

```markdown
# [文件标题]

**用途：** [一句话说明使用场景]
**状态：** 待填充

---

> 这是一个模板文件。请用你的实际内容替换以下各节的占位说明。

## [建议的章节一]
[说明这里应该写什么内容]

## [建议的章节二]
[说明这里应该写什么内容]
```

### 步骤 2.6 Skill 建议清单

按以下三个来源生成 Skill 建议，合并为一份清单：

**来源一：用户工作流分析。** 从用户描述的核心工作中，识别满足以下条件的候选：
- 每周至少做一次（高频）
- 步骤和标准相对固定（可标准化）
- 每次都需要一致的质量输出（一致性要求）

**来源二：本地已有 Skill 和提示词。** 阶段零扫描发现的 Skill 文件，已在步骤 2.4 归位。
扫描发现的提示词文件，转化为标准 SKILL.md 格式后归位。在清单中标注"已归位"。

**来源三：行业最佳实践（必须联网搜索）。** 用以下查询策略搜索：
- 搜索"[用户行业] 标准化工作流 SOP"
- 搜索"[用户角色] 常用工作模板"
- 从搜索结果中筛选可 Skill 化的工作流，补充用户可能没想到的方向

将三个来源合并，在 AGENTS.md 的"建议新增 Skills"表格中列出。

**关键动作——必须执行：** 对清单中每一个建议的新 Skill（不只是列表），
在 skills/ 目录下创建对应文件夹并生成 SKILL.md 初稿框架：

```markdown
---
name: [skill-name]
description: [触发条件和用途描述]
---

# [Skill 名称]

[一句话说明这个 Skill 做什么]

---

## 触发条件
[什么情况下使用]

## 输入
[需要用户提供什么]

## 工作流程

### 步骤一：[名称]
[待细化]

### 步骤二：[名称]
[待细化]

## 输出格式
[待定义]

## 质量标准
[待定义]
```

## 阶段三：验证 + 后续引导

### 步骤 3.1 冒烟测试

项目文件夹创建完成后，Agent 自行执行以下验证：

1. **结构完整性**：四个核心目录（context/、skills/、memory/、output/）是否存在
2. **AGENTS.md 可读性**：读取 AGENTS.md，检查三个区域（身份/规范/红线）是否都有实质内容
3. **文件索引一致性**：AGENTS.md 中列出的知识文件和 Skill 是否都在对应路径上存在
4. **CLAUDE.md 指向**：CLAUDE.md 是否正确指向 AGENTS.md

输出验证结果：

> **项目文件夹验证结果：**
> - ✅ 目录结构完整
> - ✅ AGENTS.md 三区域均有内容
> - ✅ 知识文件索引与实际文件匹配（[N]/[N]）
> - ✅ Skill 索引与实际文件匹配（[N]/[N]）
> - ✅ CLAUDE.md 正确指向 AGENTS.md
>
> 或标注具体问题：
> - ⚠️ AGENTS.md 中引用了 context/品牌/竞品分析.md，但该文件为模板未填充

### 步骤 3.2 后续引导

验证通过后，告诉用户：

> **你的项目文件夹已创建完成。接下来建议：**
>
> **现在就做（5分钟）：**
> - 打开 AGENTS.md，检查角色定义和红线是否准确，按需调整措辞
> - 把 context/ 中标注"待填充"的模板文件补上内容
>
> **使用一两周后做：**
> - 观察哪些工作在反复手动指导 AI，把它们做成 Skill
> - 发现 AI 反复犯的错误？加到 AGENTS.md 的红线或协作规范中
>
> **使用一两个月后做：**
> - 开始在 memory/MEMORY.md 中记录重要决策和工作经验
> - 做一次整体检查：有没有多余的文件？有没有缺失的？

## 冷启动路径

当用户没有现成工作目录时，跳过阶段零的扫描，改为引导式问答采集：

### 第一轮：业务定位
- 你是做什么的？（行业、具体业务）
- 你服务谁？（客户/用户画像）
- 你用 AI 主要做什么工作？

### 第二轮：工作方式
- 你最高频的工作有哪些？（每周至少做一次的）
- 有没有必须遵守的行业规范或红线？
- 你是一个人用还是团队协作？

### 第三轮：现有资产
- 你手头有没有现成的业务资料文档？（品牌介绍、产品手册、客户资料等）
- 有没有已经在用的提示词或 AI 配置？
- 有没有想让 AI "记住"的重要决策或工作原则？

三轮问完后进入阶段二，但由于没有可归位的文件，阶段二的重点是生成结构和模板。
