生成时用实际内容替换 `[方括号]` 中的占位说明。

```markdown
## 八、工作区维护协议

本章节定义 Agent 在日常协作中自动维护项目工作区的行为规范。

### 触发条件（满足任一即激活）

1. **重复纠正** — 同一行为被用户纠正 2 次以上 → 建议写入协作规范或红线
2. **Skill 候选** — 发现某项工作被反复手动指导（3 次以上）→ 建议提炼为 Skill
3. **知识缺口** — 用户在对话中提供的业务信息在 context/ 中没有对应文件 → 建议补充知识文件
4. **信息过时** — 发现 context/ 中某文件的内容与当前对话中用户提到的信息不一致 → 提醒更新
5. **结构冗余** — 某个 Skill 或知识文件超过 30 天未被引用 → 提醒审视是否需要保留
6. **决策沉淀** — 用户在对话中做出明确决策（如确认方案方向、调整协作规范、新增/移除发布渠道等）→ 建议记录到 MEMORY.md

### 行为规范

- **只建议，不擅自修改**：Agent 主动提出建议，但所有变更必须经用户确认后执行
- **建议格式**：明确说明"改什么文件""改哪个位置""为什么改""改成什么"
- **批量优先**：如果一轮对话中发现多个维护点，合并为一份建议清单，不要逐条打断用户
- **记录到 MEMORY.md**：用户确认执行的维护操作，追加到 MEMORY.md 的"工作区变更记录"中

### 边界

- 本协议不替代用户的定期增量更新（通过 witt-agent-workspace-builder 的增量更新模式执行）
- 本协议处理的是日常协作中的微调，不涉及大规模结构调整
- 如果发现需要大规模重构（如重新分类 context/ 目录），提示用户使用增量更新模式
```

### AGENTS.md 写作要点

**身份定义（第一区）**
- 3-8 行，具体且有边界
- 说明角色、专业背景、在这个项目中的定位
- 避免"你是一个有帮助的 AI 助手"这类空泛描述

**协作规范（第二区）**
- 来自用户的真实偏好
- 每条规范应该可执行（AI 能据此判断自己该怎么做）
- 如果用户在对话中反复纠正 AI 的某个行为，那条纠正就应该写在这里

**红线（第三区）**
- 优先级最高，覆盖其他所有指令
- 每条红线必须清晰、可判断
- "注意合规"是一条没有用的红线
- "不承诺具体效果或结果"是一条有用的红线

**索引表（第五、六区）**
- 让 AI 知道有哪些资源可用、什么时候该用
- "使用场景"列要写具体任务类型，不要写"需要时参考"

**工作区维护协议（第八区）**
- 这是 Agent 的"自进化"机制，让工作区在日常使用中自然迭代
- 触发条件是自动检测的，但执行必须经用户确认
- 与 witt-agent-workspace-builder 的增量更新模式互补：日常微调用第八章，大规模同步用增量更新

## 4. context/ 知识文件开头规范

每个放入 context/ 的知识文件，开头应包含以下元信息：

```markdown
# [文件标题]

**用途：** [一句话说明什么场景下参考此文件]
**最后更新：** [日期]

---

[正文内容]
```

"用途"说明帮助 AI 判断什么时候该读这个文件。
"最后更新"帮助 AI 在信息冲突时判断哪个更可信。

---

## 5. 待填充模板文件格式

对于识别出"应有但缺失"的知识文件，生成以下格式的占位模板：

```markdown
# [文件标题]

**用途：** [一句话说明使用场景]
**状态：** ⏳ 待填充
**最后更新：** [生成日期]

---

> 这是一个模板文件。请用你的实际内容替换以下各节的占位说明。

## [建议的章节一]

[这里应该写什么内容的说明，2-3句话描述要点]

## [建议的章节二]

[这里应该写什么内容的说明]
```

章节建议应该基于该文件主题的常见内容结构。
例如"品牌简介"的建议章节可能包括：企业概况、品牌理念、产品/服务体系、核心荣誉。

## 6. 目录结构命名规范

**项目根目录**：使用项目名称或业务名称，简短明确。中英文均可。

**context/ 子目录**：按信息主题分类，不按工作流分类。
- ✅ `context/品牌/`、`context/产品/`、`context/团队/`
- ❌ `context/周报用/`、`context/写方案用/`

**skills/ 子目录**：每个 Skill 一个独立文件夹，文件夹名即 Skill 名称。
- 使用小写英文 + 连字符（如 `meeting-minutes`）或中文短名（如 `会议纪要`）
- 每个文件夹至少包含一个 `SKILL.md`

**output/ 子目录**：按产出物类型分类。
- ✅ `output/方案/`、`output/报告/`、`output/脚本/`
- ❌ `output/2026-06/`（不按时间分，文件自身的命名或修改时间已包含时间信息）
